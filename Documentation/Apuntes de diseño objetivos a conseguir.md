El objetivo de estos apuntes es encontrar un tipo de juego que podamos programar el cual nos permita crear unos sistemas que fácilmente podamos expandir y que no requieran de trabajo arduo artísticamente hablando, solamente algo que pueda hacerse mediante AI. Lo importante de este proyecto no es hacer un juego, sino hacer un conjunto de herramientas y sistemas que permita crear un juego fácilmente en Unity sin necesitar de usar la escena para nada, sólo una escena bootstrap, pero que nos deje visualmente los game objects generados en la escena para debuguear (tal cual como tenemos en AutoRoguePG)
- Conseguir un método de trabajo basado en que cualquier cambio lo puede hacer la IA, encontrando maneras siempre de crear contenido con IA o procedural.
- Se debe usar siempre método de DI con Reflex en Unity, siguiendo el formato que tenemos creado en AutoRoguePG, de instalación y sistemas creados.
- Este "Framework" o modo de trabajo debe de incorporar la idea de que todo se pueda probar en runner tests que simular partes del juego que podemos probar tanto dentro del juego en modo debug, cómo tan solo ejecutándolo como test en test runner. Es muy importante que los tests que se corran en ambos sitios sean exactamente iguales al reproducirse. Este último punto es muy importante de cumplirse ya que había casos encontrados que los tests funcionaban diferente a lo que el juego luego ejecutaba en el modo escena. Validar hasta qué punto se pueden hacer tests de este tipo en cuanto involucran físicas y update frames.
- Este "Framework" que estamos preparando debe de servir para crear cualquier juego que hagamos a partir de ahora, siguiendo unas instrucciones y una idiosincrasia definida, cuanto más definida podamos hacerlo dentro de este framework que podamos tan solo instalar o trasladar como una librería como núcleo entre proyectos mejor, debiendo incluir todos los puntos detallados. Por ejemplo, cada vez que creas un sistema a traves del framework, te OBLIGA a crear test de escenarios, si es posible hacer algo así.
- La idea final de esto es poder simplemente mandar a una IA a que haga un juego siguiendo estas instrucciones y que las cumpla a rajatabla.
- Una de las cosas que no se está haciendo en AutoRoguePG que me gustaría hacer para cualquier proyecto con este "Framework" (lo podemos llamar Unidad) es que la IA debe de entender que para ciertas cosas, hay que crear sistemas más avanzados y complejos, usando técnicas de desarrollo avanzadas como las aprendidas en git-amend. Por ejemplo, en caso de incorporar IA tradicional, se debe de plantear usar un sistema propio interno que permita desarrollar y crear funciones muy expandible como EQS, creando un sistema completo de IA bien desarrollado. O por ejemplo, a la hora de crear un sistema que instancia muchos objetos, la IA automáticamente debe pensar en crear un sistema de pooling en vez de tan solo dejar la instanciación tal cual. Con esto quiero decir, que en todo punto, la IA debe entender qué sistemas habría que crear internamente como si fueran librerías propias bien desarrolladas en vez de hacer sistemas muy sencillos para salir del paso.
